# Development Roadmap: `context-crafter-mcp` from `0.6.0` baseline to first stable release

This file is a forward-looking execution plan, not a history dump.

It is intentionally grounded in the current repository state as of the tagged `0.6.0` baseline.
Anything already shipped is collapsed into a short baseline summary. The rest of this document is only about work that still needs to happen.

## 1. Product target

`context-crafter-mcp` is not aiming to be "yet another MCP server".

The target is:

> A stable, local-first, static-analysis system that can generate truthful, bounded, agent-usable repository context for real production codebases where 200k+ lines of code, deep nesting, mixed stacks, and monorepo noise are normal.

The stable release must be strong in four areas at the same time:

| Area | Stable-release bar |
|---|---|
| Truthfulness | Generated claims stay conservative under bounded scans and partial understanding |
| Scale behavior | Large, ugly repos complete without crashes and degrade honestly |
| Contract stability | CLI, MCP, validation, and machine-readable outputs are safe to depend on |
| Maintainer confidence | Release, smoke, artifact, and docs workflows are repeatable and verified |

## 2. Verified baseline at `0.6.0`

The following is already done and should not be replanned as future work:

### Product surface already present

- Python package and import layout standardized around `context-crafter-mcp` / `context_crafter_mcp`
- CLI surface exists:
  - `version`
  - `doctor`
  - `detect`
  - `generate`
  - `validate`
  - `self-test`
  - `mcp-config`
  - `serve`
- MCP stdio server exists with documented tool surface
- Generated output suite exists, including `RUN_STATE.json`
- Validation surface exists
- Resources exist for generated outputs

### Architecture and safety foundations already present

- Scanner is a real boundary: `Scanner.scan(root, options) -> RepoSnapshot`
- Deterministic scan ordering exists
- Bounded scanning exists
- Product-priority reserve exists under global scan pressure
- Output confinement exists
- Static-only analysis is the implemented model
- No target-code execution is the implemented model
- Real safety and scanner-budget tests exist

### Confidence work already present

- Fixture coverage exists for Python, Node/TypeScript, .NET, Rust, Go, Java, generic, and mixed monorepo layouts
- Real-repo smoke automation exists for Python / Node / Go / Rust
- Installed wheel and sdist smoke is already part of release verification
- `RUN_STATE.json` now includes additive-schema guidance and richer diagnostics
- Canonical scan metrics vs analyzer coverage are separated

### Release truth already established

- `0.6.0` is tagged and pushed
- Current code version is `0.6.0`
- The canonical real-repo smoke invocation is:

    uv run python scripts/smoke_repos.py

- That invocation is expected to be run from the repository root

## 3. What is not done yet

The remaining work is not basic MCP plumbing. It is stable-release hardening.

The current gaps are:

| Gap | Why it matters |
|---|---|
| Large-repo proof is still too synthetic | Stress fixtures and benchmark scripts are useful, but they are not the same as repeated proof on real 200k+ LOC repos |
| Truthfulness under severe scan pressure is not yet a finished story | Stable release must prove not just validity, but honest prioritization and clear bounded-scan signaling |
| Java and .NET confidence are still below the trust bar | They are supported, but still fixture-backed compared with Python / Node / Go / Rust |
| MCP and JSON contract freeze is not explicitly locked | Stable release must treat downstream consumers as real consumers |
| Release docs and gate execution still need to be unified around the post-`0.6.0` reality | Avoid drifting back into "docs say stable, code says maybe" territory |
| Performance budgets are not yet stated as explicit release gates | "Feels fast enough" is not enough for large-repo deployment claims |

## 4. Stable-release definition

The first stable release should be treated as `1.0.0`.

Do not cut `1.0.0` because the project already has many features.
Cut it only when the following statement is true:

> We have verified that the current scanner, analyzers, renderers, CLI, MCP server, and output contracts remain truthful and operational on large, deeply nested, mixed-stack repositories under realistic bounds, and the remaining gaps are documented rather than hidden.

## 5. Remaining workstreams

The path to `1.0.0` should be organized into six workstreams.

### Workstream A: Large-repo scale proof

Goal:

- prove that the current system is operational on large repositories, not just medium fixtures

Required outcomes:

- benchmark evidence for 200k+ LOC or similarly heavy file counts
- repeated runs on deep, ugly trees with vendor/generated/test noise
- explicit timing and memory expectations for stable release
- documented failure/degradation modes when limits are hit

Concrete tasks:

1. Define a release-scale benchmark matrix
   - synthetic large tree
   - deep mixed-stack synthetic tree
   - at least one large Python repo
   - at least one large Node/TypeScript monorepo
   - at least one large polyglot or services-style repo
2. Extend benchmarking beyond `scripts/bench_scan.py`
   - record scan time
   - record full pipeline time
   - record peak memory if practical
   - capture whether bounds were exhausted
3. Add repeatable challenge-repo documentation
   - exact commands
   - expected output characteristics
   - what counts as acceptable bounded degradation
4. Decide explicit stable-release performance gates
   - scanner time budget
   - pipeline time budget
   - memory ceiling
   - acceptable bounded-scan warning behavior

Release bar for Workstream A:

- stable release has written, verified evidence that the tool remains usable on repositories much larger and messier than the fixture set

### Workstream B: Truthfulness under bounded scans

Goal:

- ensure that when the tool cannot fully understand a large repo, it says so clearly and still prioritizes the most important surfaces

Required outcomes:

- bounded-scan warnings remain visible in every relevant output surface
- primary conclusions continue to prioritize product code over vendor/generated/tooling noise
- unknown/unsupported/error evidence stays conservative rather than optimistic

Concrete tasks:

1. Build more adversarial large-repo tests
   - deep nesting with multiple plausible roots
   - monorepo with many low-signal packages and a few critical entrypoints
   - repos where docs/tooling outnumber product files
   - generated-output-heavy repos
2. Audit rendered outputs for truth drift
   - `PROJECT_OVERVIEW.md`
   - `ARCHITECTURE_SUMMARY.md`
   - `AGENT_BRIEF.md`
   - `SCAN_REPORT.md`
   - `RUN_STATE.json`
3. Add assertions for "what must not happen"
   - vendor tree becomes the main architecture story
   - generated code becomes the main entrypoint claim
   - bounded scan is hidden from the brief summary
   - analyzer parse count is misread as scanner truth
4. Decide whether more prioritization/ranking work is still needed
   - only if tests show real failures
   - not as speculative rewrite work

Release bar for Workstream B:

- challenge-repo and scale tests show honest summaries even when the repo is partially scanned or partially parsed

### Workstream C: Stack confidence hardening

Goal:

- make stable-release support claims line up with actual verified depth per language

Current reality:

- Python / Node / Go / Rust: meaningful real-repo smoke confidence
- Java / .NET: still lower-confidence and fixture-backed

Concrete tasks:

1. Freeze the public confidence language by stack
   - what is stable
   - what is supported but lower-confidence
   - what is intentionally heuristic
2. Raise Java / .NET confidence enough for stable-release honesty
   - manual validation on real repos, or
   - stronger challenge fixtures plus clearly documented limitations
3. Re-check parser fallback behavior
   - parser installed
   - parser missing
   - parse failures
   - mixed parser/fallback evidence
4. Decide whether any stack claim in README/docs should be narrowed before `1.0.0`

Release bar for Workstream C:

- every supported stack has an explicit trust statement backed by either smoke evidence or clearly documented limitation

### Workstream D: Contract freeze and consumer safety

Goal:

- make the CLI, MCP, validation, and machine-readable outputs safe for downstream automation to depend on

Required outcomes:

- tool names and signatures are treated as intentionally stable
- `RUN_STATE.json` additive contract is enforced in practice
- generation/validation JSON result shapes are treated as compatibility surfaces

Concrete tasks:

1. Review CLI contract stability
   - flags
   - defaults
   - JSON output fields
   - exit-code behavior
2. Review MCP surface stability
   - tool names
   - argument names
   - response structure
   - resources URI scheme
3. Add regression coverage for consumer-facing fields
   - especially `RUN_STATE.json`
   - generation JSON
   - validation JSON
   - capability descriptions if externally consumed
4. Decide what remains explicitly non-stable before `1.0.0`
   - document it
   - do not leave it implicit

Release bar for Workstream D:

- a downstream maintainer can rely on the machine-readable surfaces without fearing silent shape drift

### Workstream E: Release and docs truth discipline

Goal:

- keep the release line and roadmap truthful while the project moves from hardening into stable release

Concrete tasks:

1. Keep release-state docs synchronized whenever the release state changes
   - `CHANGELOG.md`
   - `docs/ROADMAP.md`
   - `docs/REAL_REPO_SMOKE_MATRIX.md`
   - `docs/project_state.md`
   - `IMPLEMENTATION_REPORT.md`
2. Rewrite roadmap docs so they describe remaining work, not already-finished milestones
3. Make sure large-repo aspirations are written as verified or planned, never assumed
4. Keep the canonical smoke, artifact, and release commands exact

Release bar for Workstream E:

- no release-facing doc materially overstates what has been verified

### Workstream F: Execute the stable-release gate

Goal:

- run the real `1.0.0` gate end-to-end on a clean checkout

This workstream should happen after A through E are sufficiently complete.

Concrete tasks:

1. Update `docs/PRE_1_0_GATE.md` if needed to reflect the final release truth
2. Run the full gate on a clean checkout
3. Re-run wheel and sdist verification
4. Re-run real-repo smoke verification
5. Confirm docs match the actual stable surface
6. Only then bump version to `1.0.0`
7. Tag and publish the stable release

Release bar for Workstream F:

- the gate is executed, not assumed

## 6. Execution order

Do the remaining work in this order:

1. Workstream E first where docs are stale enough to mislead current work
2. Workstream A to establish real scale evidence
3. Workstream B to harden truthfulness under that evidence
4. Workstream C to lock honest stack confidence
5. Workstream D to freeze consumer-facing contracts
6. Workstream F to execute the stable-release gate

Reason:

- there is no value in freezing contracts or cutting `1.0.0` before the large-repo truthfulness story is proven

## 7. Concrete plan by milestone

### Milestone 1: post-`0.6.0` truth reset

Deliverables:

- all release-state docs reflect that `0.6.0` is already public
- this roadmap is rewritten around remaining work only
- `docs/ROADMAP.md` and `docs/PRE_1_0_GATE.md` are aligned with the stable-release direction

Exit criteria:

- no roadmap/release doc still talks as if `0.6.0` is pending

### Milestone 2: scale and challenge-repo evidence

Deliverables:

- repeatable benchmark/challenge suite
- explicit performance observations
- documented large-repo expectations

Exit criteria:

- the project has hard evidence, not just intuition, for large-repo behavior

### Milestone 3: truthfulness hardening under pressure

Deliverables:

- new adversarial tests
- output reviews converted into regression coverage where practical
- bounded-scan truthfulness accepted as good enough or further targeted fixes identified

Exit criteria:

- large mixed repos do not silently produce overconfident summaries

### Milestone 4: stack confidence freeze

Deliverables:

- final stable-release support wording by language
- Java / .NET confidence decision documented
- parser/fallback expectations verified

Exit criteria:

- public stack support claims are explicit, stable, and justified

### Milestone 5: contract freeze

Deliverables:

- stable CLI/MCP/result-surface review
- additive-only machine-readable contract policy enforced by tests/docs

Exit criteria:

- downstream consumers can safely automate against the current surfaces

### Milestone 6: `1.0.0` gate and release

Deliverables:

- clean checkout gate execution record
- final docs truth audit
- `1.0.0` tag

Exit criteria:

- the stable release is a verified event, not just a version bump

## 9. What not to do

Avoid these traps while chasing stable release:

| Trap | Why it is a trap |
|---|---|
| Rewriting the scanner in Rust/Go before proof demands it | It burns time before the current Python boundary is actually exhausted |
| Expanding into cloud/API/LLM features | It weakens the deterministic local-first product story |
| Treating green fixtures as proof of large-repo readiness | Stable release depends on challenge-repo and scale evidence too |
| Cutting `1.0.0` because the CLI/MCP surface feels complete | Stable release is about trust under pressure, not feature count |
| Hiding heuristic behavior behind polished prose | It makes the output less safe in the exact codebases this tool targets |
| Growing the MCP surface without a strong reason | It increases consumer risk and tool-selection noise |

## 10. Final release rule

Release `1.0.0` only when all of the following are true:

- `0.6.0`-style hardening has been extended into verified large-repo proof
- bounded-scan truthfulness is good enough for ugly real codebases
- stack confidence claims are honest and frozen
- CLI, MCP, validation, and `RUN_STATE.json` contracts are intentionally stable
- release docs match reality
- the explicit stable-release gate has been executed end-to-end

If any one of those is false, do another hardening release instead of forcing `1.0.0`.
