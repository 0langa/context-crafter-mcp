## 1. Final Product Shape

The final project is both:

| Surface | Purpose |
|---|---|
| MCP server | Main product surface for coding clients and AI agents |
| CLI app | Human/admin/debug surface for setup, diagnostics, manual generation, and config snippets |

Product statement:

> Context Crafter MCP turns source repositories into compact, source-grounded AI-agent context through a local-first MCP server and CLI.

The v1 server is stdio-first. Docker and HTTP service modes are optional future work, not default v1 requirements.

Required CLI surface:

    context-crafter-mcp --help

## 2. Architecture Decision

Final decision:

> Keep Python for the MCP server, CLI, analyzer registry, renderers, validation reports, and package workflow. Design the filesystem scanner as a strict replaceable boundary. Add Tree-sitter for deeper parsing before considering a lower-level scanner. Only add Rust/Go after benchmarks prove Python traversal is the bottleneck.

Rationale:

| Component | Decision | Rationale |
|---|---|---|
| MCP server | Python | Python is a first-class MCP SDK path and fits tool/resource/prompt servers |
| CLI | Python | Text generation, JSON output, config generation, and Windows support are straightforward |
| Analyzer registry | Python | Registry logic depends on contracts, models, fixtures, and tests more than raw speed |
| Markdown/HTML rendering | Python | Rendering and validation are not performance-critical |
| Safe filesystem scanner | Python first, replaceable later | Lower-level may help later, but only after benchmarks prove it |
| Deep parsing | Tree-sitter first | Better correctness upgrade than rewriting the whole project |
| Future scanner engine | Optional Rust/Go helper | Only if benchmark thresholds are crossed |

Required ADR:

    docs/adr/0001-python-product-layer-replaceable-scanner.md

Required scanner boundary:

    Scanner.scan(root, options) -> RepoSnapshot

Everything above the scanner must consume `RepoSnapshot`, not raw ad-hoc filesystem traversal.

## 3. Current State Assessment

The project already has a useful core idea and should stay public if active, but it needs productization and proof before being a strong GitHub pin.

Current strengths:

| Area | Current state | Why it matters |
|---|---|---|
| Product idea | Local-first MCP server that generates repo docs/context | Strong CV signal for AI tooling and developer automation |
| Structure | Python package, tests, workflow, `pyproject.toml`, lockfile | Already beyond a throwaway script |
| Supported stacks | Python, Node/JS, .NET, Rust, Go, Java, generic | Good breadth if proven by fixtures |
| Safety story | Static-only, bounded scan, no target execution | Excellent if backed by tests |
| Tool surface | Detection and doc generation tools | Good base, but needs cleaner contracts |
| Weak point | Early-stage public polish | Needs examples, install path, tests, docs, release story |

Main gaps:

| Gap | Fix |
|---|---|
| Name/branding still transitioning | Standardize local package/repo/docs around `context-crafter-mcp` |
| Python choice could look accidental | Add ADR and scanner boundary |
| README not product-grade enough | Rewrite for quick understanding and proof |
| No visible generated examples | Commit `examples/demo-repo` and `examples/outputs` |
| Broad language claims need proof | Add fixture repos and golden tests |
| MCP behavior needs proof | Add Inspector docs and JSON-RPC/SDK smoke tests |
| Safety claims need proof | Add adversarial scanner/generation tests |
| Output contract needs stabilization | Define generated files and metadata |
| Release/install flow needs polish | Add console scripts, `uvx`/`pipx` docs, build checks |

## 4. Non-Goals

| Non-goal | Reason |
|---|---|
| Full documentation website generator | Dilutes MCP/context focus |
| API-key/LLM generation | Weakens deterministic local-first story |
| Executing target project code | Unsafe and unnecessary for static analysis |
| Rust/Go rewrite | Premature unless benchmark gates fail |
| Docker-first deployment | Unnecessary friction for v1 coding-client use |
| Remote daemon/service mode | Not needed for local stdio MCP v1 |
| Automatic publishing | Requires maintainer review |
| Automatic remote repo rename | Manual GitHub action |

| Requirement | Bar |

| Maintained | Changelog, license, contributing, security, release notes exist |

| `include_hidden` | false |
| `include_tests` | true |
| `max_files` | bounded |
| `max_depth` | bounded |
| `max_file_bytes` | bounded |

Required tests:

| Test | Expected behavior |
|---|---|
| symlink loop | skipped/terminated safely |
| binary file | skipped |
| huge file | skipped |
| invalid encoding | reported without crash |
| ignored dir | skipped |
| max files | graceful warning |
| max depth | graceful warning |
| deterministic order | stable output |

Benchmark gate:

    Do not build Rust/Go scanner unless:
      - 100k-file scan with ignore rules takes >5 seconds on a normal dev machine, or
      - memory exceeds 250 MB, or
      - scanner complexity becomes harder to reason about than a small helper.

## 9. What Not To Do

Avoid:

| Trap | Why |
|---|---|
| rewriting in Rust/Go now | slows product progress without solving current UX/docs/test gaps |
| adding LLM/API features | dilutes deterministic local-first story |
| claiming deep understanding | static analysis must remain honest |
| publishing packages | requires maintainer review |
| remote repo rename by agent | manual GitHub action |
| Docker-first workflow | unnecessary friction for v1 |
| too many MCP tools | hurts model tool choice |
| generated docs without tests | output quality becomes untrustworthy |
| README claims not backed by code | hurts CV-facing credibility |
| messy duplicate package trees | creates maintenance debt |
| placeholder TODO implementations | creates false completion |
